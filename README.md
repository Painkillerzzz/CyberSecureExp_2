# CyberSecureExp_2

# 实验二：基于口令的安全身份认证协议

## 实验目的

口令是常见的身份认证方式，用户通过键入身份标识和口令来证明其身份的合法性与真实性。目前京东、淘宝等知名电商网站均保留了这种认证方式。然而，如果在不安全的信道环境当中直接发送口令，口令将被窃听者直接窃取，进而实施身份伪造。此外，在认证结束后，如果直接使用口令当作对称式加密通讯的密钥，在多次通信中将会因密钥重用产生相同密文，这就给了攻击者发起重放攻击的可乘之机。  
Bellovin-Merritt 协议诞生在贝尔实验室，相关文章发表在 1992 年的 Oakland 会议上，是最早的基于口令的安全身份认证协议。这一协议最早实现了在不安全信道上通过口令完成双方身份认证，并在认证结束后使通信双方共享安全密钥，也就是在身份认证的同时完成密钥分发。解决了不安全信道中口令被窃听的问题，以及直接利用口令作为密钥产生的消息重放问题。该协议设计简单，开创了安全口令认证协议的一系列研究，并被广泛地认为是安全的。本实验将尝试实现 Bellovin-Merritt 身份认证协议。这一协议将传统的对称、非对称加密算法作为子模块使用，通过对基础密码算法库的调用可以加深对公钥密码、对称密码、散列函数工作方式的理解，并且有助于熟悉密码学编程接口。这一身份认证协议的实现过程中涉及大整数操作，通过编码实现这一协议还可以体验 OpenSSL 等安全协议库的实现流程。

## 实验环境设置

本实验需要一台配备有 Java JDK 的主机，操作系统环境不限（推荐 1.8 以上的高版本 JDK）。因为该协议当中涉及大整数操作，推荐使用 Java 实现 这一协议。Java 当中配备有完备的大整数运算库，并且 java.security 包当中包 含了大量的成熟的经典加密算法（例如 AES、DES、RSA），可用于认证协议 的实现，且 Java 运行效率较高，能负担认证协议算法的计算开销。

## 实验步骤

下面介绍 Bellovin-Merritt 协议，其流程如图 6 所示。这个协议用到两个对 称加密方案和一个公钥加密方案作为基本模块，两个对称加密方案的加密算法 分别记为 $E_0$ 和 $E_1$，公钥加密算法记为 $E$。

 - 初始化：$A$、$B$ 为两个通信实体，事先共享口令 $pw$，$A$ 希望通过向 $B$ 证明其知晓口令 $pw$ 来认证其身份的合法性与真实性。 
 - 步骤 1：$A$ 每次开始与 $B$ 认证时，随机生成一对新的公钥和私钥 $(pk_A, sk_A)$ 用于公钥加密方案 $E$。然后，向 $B$ 发送自己的身份标识以及用 $pw$ 加密后的 $pk_A$ 的密文。
 - 步骤 2：$B$ 接收到该消息后，以 $pw$ 解密密文得出 $pk_A$。然后随机生成会 话密钥 $K_s$，并把经过两重加密后的密文 $E_0(pw, E(pk_A, K_s))$ 发送给 $A$。
 - 步骤 3：$A$ 接收到消息后用 $pw$ 和 $sk_A$ 解密得到 $K_s$，并生成随机数 $N_A$， 用 $K_s$ 加密 $N_A$ 并将密文 $E_1(K_s, N_A)$ 发送给 $B$。
 - 步骤 4：$B$ 接到消息后用 $K_s$ 解密得出 $N_A$，并生成随机数 $N_B$，然后以 $K_s$ 加密 $N_A||N_B$（$N_A$ 和 $N_B$ 的二进制表示相拼接），并将密文 $E_1(K_s, N_A||N_B)$ 发送回 $A$。
 - 步骤 5：$A$ 在接收到消息后以用 $K_s$ 解密得两个随机数的明文：$N = N_1||N_2$， 验证其中的随机数 $N_1$ 和 $N_A$ 是否相等，如果相等则判定对方确实是 $B$，接下 来用 $K_s$ 加密另一个随机数 $N_2$，并将密文 $E_1(K_s, N_2)$ 发送到 $B$。
 - 步骤 6：$B$ 接到消息后用 $K_s$ 解密得出 $N_2$，并验证 $N_2$ 是否和 $N_B$ 相等， 如果相等则判定对方确实是 $A$，并且认证结束，然后 $A$ 和 $B$ 采用 $K_s$ 作为共 享的对称式密钥通信。

下面介绍实现当中的一些细节。可以选用 RSA 作为公钥加密算法 $E$ 的实现，AES 和 DES 分别为对称加密方案的加密算法 $E_0$ 和 $E_1$ 的实现。实现的协议应当包含两个进程分别模拟通信双方 $A$ 和 $B$，两个进程通过 Java 进程间通信模块或套接字联网 API 交互协议报文。两进程运行前，应当向两个进程分配共享口令。之后进程应进行 Base64 编码，并利用 Java 密码学算法库中的散列函数获得其散列，因为用户的口令长度为不定，无法满足某些密码算法操作的要求。$A$ 进程应调用 Java 密码学库下的工业级 RSA 模块，生成随机公钥和私钥对，并进行相应的非对称加解密操作。同理使用 Java 的 Security 包下的 AES 和 DES 算法作对称式加密。

## 预期实验结果
当实现认证双方 $A$、$B$ 的进程后，$A$ 在运行时向 $B$ 发起认证请求。双方
进行多轮交互后，$A$ 和 $B$ 完成双向的身份认证，$A$ 确认 $B$ 持有口令 $pw$，并
且 $B$ 通过认证协议确定 $A$ 的身份为真实。之后 $A$ 和 $B$ 将得到相同的共享密
钥 $K_s$，并且可用其作为之后通信的对称式加密的密钥。

## 代码实现说明

本仓库已给出一个基于 Java 的 Bellovin-Merritt 协议完整实现，使用 Maven 管理工程，主要结构如下：

- `pom.xml`：Maven 工程与 JDK 1.8 编译配置。
- `src/main/java/edu/cybersecure/exp2/CryptoUtils.java`：密码学基础函数封装，用于：
  - 通过“Base64 编码 + SHA-256 散列”从口令派生 AES 密钥，实现 $E_0$；
  - 从会话密钥 $K_s$ 派生 DES 密钥，实现 $E_1$；
  - 提供 AES/ECB/PKCS5Padding 与 DES/ECB/PKCS5Padding 的加解密接口；
  - 生成随机字节序列，用于会话密钥与随机数 $N_A$、$N_B$。
- `src/main/java/edu/cybersecure/exp2/Messages.java`：定义协议交互报文类型 `Msg1`~`Msg5`，均实现 `Serializable`，通过 Java Socket 的对象流进行进程间通信。
- `src/main/java/edu/cybersecure/exp2/Alice.java`：模拟实体 $A$：
  - 派生口令 AES 密钥；
  - 运行时生成一次性 RSA 公私钥对 $(pk_A, sk_A)$；
  - 按实验步骤 1~5 实现 Bellovin-Merritt 协议中的 A 端逻辑；
  - 与 B 完成双向认证后，得到共享会话密钥 $K_s$。
- `src/main/java/edu/cybersecure/exp2/Bob.java`：模拟实体 $B$：
  - 监听指定端口，等待 A 连接；
  - 使用共享口令恢复 $pk_A$，生成会话密钥 $K_s$ 并完成协议步骤 2、4、6；
  - 验证随机数一致性，完成对 A 的身份认证，并最终与 A 共享密钥 $K_s$。

上述实现中：

- 公钥加密算法 $E$ 使用 Java 标准库中的 RSA（`RSA/ECB/PKCS1Padding`）；
- 对称加密算法 $E_0$ 使用 AES（`AES/ECB/PKCS5Padding`），密钥由共享口令经 Base64 编码和 SHA-256 散列后截断得到；
- 对称加密算法 $E_1$ 使用 DES（`DES/ECB/PKCS5Padding`），密钥由会话密钥 $K_s$ 的前 8 字节派生；
- $A$ 与 $B$ 使用 Java Socket (`ServerSocket`/`Socket`) 及 `ObjectInputStream`/`ObjectOutputStream` 实现进程间通信。

## 编译与运行

1. **准备环境**
   - 安装 JDK 1.8 及以上版本；
   - 安装 Maven（3.6+ 即可）。

2. **编译工程**

   在仓库根目录（本文件所在目录）执行：

   ```bash
   mvn clean package
   ```

   成功后将在 `target/` 目录下生成可运行的 jar 包（例如 `CyberSecureExp_2-1.0-SNAPSHOT.jar`）。

3. **运行实体 B（服务器）**

   在一终端窗口中运行（假设口令为 `mypassword`，监听 5000 端口）：

   ```bash
   java -cp target/CyberSecureExp_2-1.0-SNAPSHOT.jar edu.cybersecure.exp2.Bob 5000 mypassword
   ```

4. **运行实体 A（客户端）**

   在另一终端窗口中运行（假设 B 在本机 127.0.0.1，端口 5000，口令需与 B 完全一致）：

   ```bash
   java -cp target/CyberSecureExp_2-1.0-SNAPSHOT.jar edu.cybersecure.exp2.Alice 127.0.0.1 5000 mypassword
   ```

5. **预期输出**

- B 端会输出：
  - 收到 A 的身份标识；
  - 成功解密得到 `pk_A`；
  - 发送和接收各轮密文；
  - 最终提示“验证 Nb 成功，确认对方为 A。双向认证完成！”等信息。
- A 端会输出：
  - 生成临时 RSA 密钥对；
  - 成功解密得到会话密钥 `Ks`；
  - 验证 `Na` 成功；
  - 最终提示“双向认证完成，A 与 B 已共享会话密钥 Ks，可用于后续对称加密通信。”等信息。

这样即完整重现实验文档中描述的 Bellovin-Merritt 基于口令的安全身份认证协议，并在实验结束时获得共享会话密钥 $K_s$。

## 异常情况与认证终止说明

在实际运行过程中，若出现以下情况，程序会打印提示并“认证终止”，便于实验观察和调试：

- **口令不一致或报文被篡改**
  - B 端使用共享口令解密 `pk_A` 失败：`[B] 使用口令解密 pk_A 失败，可能口令不一致或报文被篡改，认证终止。`
  - A 端使用口令解密会话密钥 `Ks` 失败：`[A] 使用口令解密会话密钥失败，可能口令不一致或报文被篡改，认证终止。`

- **公钥 / 会话密钥格式或解密异常**
  - B 端无法将解密后的 `pk_A` 解析成合法 RSA 公钥：`[B] pk_A 格式非法，认证终止。`
  - A 端使用私钥对 `E(pk_A, Ks)` 解密失败：`[A] 使用私钥解密会话密钥失败，认证终止。`

- **会话阶段的加解密失败或消息结构异常**
  - B 端或 A 端使用 `Ks` 解密 `Na`、`Na||Nb`、`Nb` 时出错（如密钥不一致、密文被截断或篡改），会分别输出“使用 Ks 解密 Na/Na||Nb/Nb 失败，认证终止。”等信息。
  - A 端拆分 `Na||Nb` 时发现长度不是偶数，无法一分为二：`[A] 收到的 Na||Nb 长度异常，认证终止。`
  - A 端验证 `Na` 失败：`[A] 验证 Na 失败，认证终止。`
  - B 端最终验证 `Nb` 失败：`[B] 验证 Nb 失败，认证终止。`

- **网络 / 通信异常**
  - B 端接收不到来自 A 的首个报文或后续包含 `Na`、`Nb` 的报文：打印“接收来自 A 的首个报文失败 / 接收包含 Na/Nb 的报文失败，认证终止。”
  - A 端收不到包含会话密钥的响应或包含 `Na||Nb` 的响应：打印“未能收到包含会话密钥的响应 / 未能收到包含 Na||Nb 的响应，认证终止。”

通过上述设计，协议在检测到任意异常或潜在攻击迹象时会立即停止，不会继续在不可信状态下运行。
